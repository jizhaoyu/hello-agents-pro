## 1.1 什么是智能体?

### 🎯 教科书定义(四个基本要素 + 一个灵魂)

> **智能体(Agent)= 能通过传感器(Sensors)感知环境(Environment),并自主地通过执行器(Actuators)采取行动(Action)以达成特定目标(Goal)的实体。**

| 要素 | 含义 | 自动驾驶的例子 | 类比 |
|------|------|---------------|------|
| **环境 Environment** | 它活在什么世界里 | 动态变化的道路交通 | 你下棋的棋盘 |
| **传感器 Sensors** | 怎么"看"世界 | 摄像头、雷达、GPS | 你的眼睛和耳朵 |
| **执行器 Actuators** | 怎么"动"世界 | 方向盘、油门、刹车 | 你的手和脚 |
| **目标 Goal** | 它要达到什么 | 安全快速地到达目的地 | 你要赢这盘棋 |

**灵魂 = 自主性(Autonomy)**:智能体不是提线木偶。它不是被动响应刺激,而是**基于自己的感知和内部状态,独立做决策**。

> 🔑 **关键洞察:自主性不是"有/无"的开关,而是一条光谱(spectrum)。** 恒温器在最底端(if 温度>设定→制冷),自动驾驶在高端(实时感知+规划+权衡+学习)。**它们都叫"智能体",差别只是在这条光谱上站多高。**

画成图就是一个闭环:

```
   ┌──────────────────────────────────────┐
   │                                      │
   ▼                                      │
[传感器感知] ──→ [自主决策] ──→ [执行器行动]
   │                ↑                │
   └──────────────────────────────────┘
              环境反馈
```

> 🧊 **这个闭环是全书最重要的图。** 每一章的"新范式",本质上都是在某一步上做了升级:更好的感知→更聪明的决策→更精准的行动→更完整的反馈。万变不离这个圈。

### 🧭 三把分类的尺(三个独立的坐标轴)

#### 📐 第一把尺:内部架构复杂度(1.1.1 传统智能体的五级台阶)

这是最经典的分类。每上一级,就回答一个前一级答不了的新问题,**自主性+1**:

| 台阶 | 类型 | 它回答的新问题 | 类比 | 经典例子 |
|------|------|---------------|------|---------|
| **①** | **简单反射** Simple Reflex | "现在该干嘛?"(只看当下这一刻) | 膝跳反射:敲一下就弹,不问原因 | 恒温器:温度>设定→制冷 |
| **②** | **+世界模型** Model-Based | "**世界现在到底什么样?**"(含我看不见的) | 隧道里开车:眼睛看不见前车,**但脑子记得它还在前面** | 自动驾驶在隧道中仍维持对前车位置/速度的判断 |
| **③** | **+目标** Goal-Based | "怎么做才能**达成目标**?"(开始规划未来) | GPS 导航:不只是"看到路",而是**主动搜索最优路线** | GPS + A* 算法规划路径 |
| **④** | **+效用** Utility-Based | "哪种做法**最满意**?"(权衡冲突目标) | 不只想到公司,还要最快+最省油+避拥堵,**三件事一起权衡** | 多目标路径优化 |
| **⑤** | **+学习** Learning | "能不靠人预设,**自己从经验里学**吗?" | 下棋AI:开始乱下→赢了加分→几百万局后自己悟出棋路 | AlphaGo Zero:从零自弈,发现人类未见过的策略 |

> 🔴 **灵魂红线:①~④ 不管多复杂,本事归根结底是人"灌"进去的(规则/模型/效用函数,全由工程师设计)。只有到了⑤学习型,智能体才第一次挣脱了"人喂什么才会什么"的枷锁——它能靠跟环境互动自己长本事。**

AlphaGo Zero 能下出超越人类几千年的棋路,不是因为它背了更多棋谱,**而是它通过自我对弈,自己发现了人类从未走过的有效策略。** 这就是"学习"二字的真正分量。

---

#### 📐 第二把尺:决策时间(反应 vs 规划)

这把尺问的是:**智能体是"想都不想就动",还是"想清楚了再动"?**

| | ⚡ 反应式 Reactive | 🧠 规划式 Deliberative |
|---|---|---|
| **怎么决策** | 感知→行动 直接映射,不绕弯 | 行动前先用世界模型推演未来、评估后果 |
| **类比** | 条件反射:手摸到烫的立刻缩 | 棋手:算后面十几步再落子 |
| **优势** | **快、省、稳**——毫秒级响应 | **有远见、战略性**——能处理复杂多步任务 |
| **代价** | **短视**——只看眼前一步,易陷局部最优 | **慢、贵**——算得久,且环境可能在你算的时候已经变了 |
| **例子** | 安全气囊(碰撞 0.0x 秒内必弹出) | GPS 规划路线/制定商业计划 |

现实任务往往**两个都要**:旅行助手既要能即时回应"这家酒店太贵了!"(反应),又要能规划出为期数天的完整行程(规划)。所以有了——

**🔀 混合式 Hybrid**:底层放一个快速反应模块(处理紧急/基本动作),上层放一个审慎规划模块(负责长远目标)。**而 LLM 智能体,就是一种最灵活的混合式实现。**

它在 **「思考→行动→观察」** 微循环中巧妙融合两者:
- **思考(Reasoning)** = 审议/规划(慢思考)
- **行动+观察(Acting & Observing)** = 反应(快反馈)

> 🎯 一个宏大任务,被拆成一连串"规划-反应"微循环——既灵活应对即时变化,又通过连贯步骤最终完成长期目标。**这个"思考→行动→观察"循环,就是第四章 ReAct 范式的灵魂。**

---

#### 📐 第三把尺:知识表示(符号 vs 神经)——最深的一刀

这把尺问的是:**智能体脑子里的"知识",到底长什么样?** 这是 AI 领域吵了半个多世纪的根基问题,劈出了两种截然不同的"文化"。

| | 🗂️ 符号主义 Symbolic AI | 🧒 亚符号主义 Sub-symbolic AI(连接主义) |
|---|---|---|
| **知识存哪** | 显式的规则库、知识图谱(人可读) | **隐式地、分布式地**存在神经网络的连接权重中(人不可读) |
| **怎么获得** | 工程师+领域专家一条条手工编写 | 从海量数据中**自己学出**统计模式 |
| **类比** | 👨‍💼 **图书管理员**:把世界知识梳理成清清楚楚的目录和索引 | 👶 **学说话的孩童**:不是靠背"猫=四条腿+毛茸茸+喵喵叫",而是看过成千上万张猫图后,**大脑自己辨识出了"猫"那个模式** |
| **优势** | **透明、可解释**——每一步推理都看得见,在高风险领域(金融/医疗)是刚需 | **模式识别强、抗噪声鲁棒**——图像/声音/文本轻松拿下,能找到人发现不了的规律 |
| **致命伤** | **脆弱**——遇到规则外的情况立刻失灵;**知识获取瓶颈**——人的直觉性知识根本写不成 IF-THEN | **黑箱**——能高准确率认出猫,但问"你凭什么说这是猫?"→答不上来;**纯逻辑推理弱+会一本正经地胡说(幻觉)** |

> 🧩 看出来了吗?符号主义强在推理但弱在感知,连接主义强在感知但弱在推理。**两家的优点正好互补,缺点正好是对方的长处。** 所以第三条路呼之欲出——

**🤝 神经符号主义 Neuro-Symbolic AI**:把两个冤家揉成一个系统,既能像神经网络那样从数据里学直觉,又能像符号系统那样做逻辑推理。

书里用诺奖得主卡尼曼的《思考,快与慢》双系统来做绝妙类比:

| 卡尼曼的理论 | 思维特征 | 对应的AI流派 |
|---|---|---|
| **系统1(快思考)** | 快速、凭直觉、不费力、并行 | 亚符号主义——神经网络模式识别 |
| **系统2(慢思考)** | 缓慢、讲逻辑、费脑力、审慎 | 符号主义——逻辑推理 |

> ⚠️ **勘误**:书里把系统1/系统2的**编号写反了**(说"系统1是缓慢逻辑思维")。正确版本:**系统1=快/直觉(↔亚符号),系统2=慢/逻辑(↔符号)**。概念配对是对的,编号错了——这种地方正是"理解原理"比"背书"值钱的地方。

---

### 🚀 LLM 智能体 = "新范式",不是在老阶梯上又爬高一级

前面已经搭好了三把尺,现在我们把 LLM 智能体放上去量一下:

**它是什么?** LLM 智能体的**内核是一张巨大的神经网络**(→ 亚符号/快思考的底子),但当它工作时,**会吐出一串结构化的中间步骤——Thought(思考)、Plan(计划)、API Call(工具调用)**——这些都是明确、可操作的**符号**(→ 符号/慢思考)。

> 🎯 **关键洞察:在一个系统里,神经网络负责"生成",它生成出来的东西恰好就是符号。** "生成即符号化"——这就是神经符号主义在工程上最精妙的落地。

**它的知识从哪来?** 不是在老阶梯上从④爬到⑤,而是**换了造阶梯的材料**:
- 老阶梯①~④:知识=人手工编码的规则和模型
- LLM 智能体:知识=预训练在海量数据里**自己泡出来的隐式世界模型**

**它的能力边界在哪?** 它不是阶梯上某"一级",而是**同时干了②(世界模型)+③(规划)+④(权衡)**,哪一级都对得上,又哪一级都装不下。

**最终定位**:

```
LLM 智能体 = 混合式(反应+规划) × 神经符号(神经网络底子+符号化输出)
           = 通用大脑 + 自然语言引导 + "思考→行动→观察"循环
```

> 📌 这句话会在第四~六章反复被印证:ReAct、Plan-Solve、Reflection、LangGraph,本质上都是"怎么在工程上把这个循环组织得更聪明"的不同答案。

## 1.2 智能体的构成与运行原理

上一节把智能体"是什么"讲透了。这一节回答:**它怎么工作?**

### 🧱 1.2.1 PEAS 模型:开工前先把活说清楚

动手造智能体之前,工程师会用 **PEAS** 把任务环境定义清楚:

| 字母 | 含义 | 问的是 | 旅行助手例子 |
|------|------|--------|------------|
| **P** | Performance 性能度量 | 怎么算干得好? | 行程合理、不超预算、用户满意 |
| **E** | Environment 环境 | 活在什么世界里? | 天气API/地图API/订票API 构成的数字世界 |
| **A** | Actuators 执行器 | 能动什么? | 调用各类工具(查天气/搜景点/订票…) |
| **S** | Sensors 传感器 | 怎么接收信息? | API返回的数据、用户的文本输入 |

LLM 智能体所处的数字环境有**四个"难搞"特性**,这些特性直接决定了后面章节要解决的工程问题:

| 环境特性 | 说的是 | 逼出什么能力 | 后面哪章解决 |
|---------|--------|------------|------------|
| **部分可观察** Partially Observable | 你永远看不到全局。查机票时拿不到所有航司的全部实时数据,只能看到你调的那个 API 返回的那一丁点 | **记忆**(记住查过什么)+**探索**(换着法子查) | 第八章 记忆系统 |
| **随机性** Stochastic | 同一动作做两次,结果可能不同。相邻两次搜机票,价格和余票都可能变了 | **处理不确定性**,监控变化、及时决策 | Agent Loop 的天然设计 |
| **多智能体** Multi-agent | 环境里还有别人在抢。别的用户、动态调价系统,都在跟你竞争同一张特价票 | **快速响应**,策略动态调整 | 第六/十/十三~十五章 多智能体 |
| **序贯且动态** Sequential & Dynamic | 你现在的动作影响未来,且你思考时环境还在变 | 循环必须**又快又灵活** | 核心 Agent Loop |

---

### 🔁 1.2.2 Agent Loop:智能体的"心跳"

智能体不是一次性的输入→输出,而是**靠一个持续的循环不断转圈**:

```
         ┌─────────────────────────────────────────────┐
         │                                             │
    [① 感知 Perception]                        [③ 行动 Action]
     接收 Observation                            调工具/执行动作
     (用户指令 or 上一步反馈)                      对环境施加影响
         │                                             │
         ↓                                             ↓
    [② 思考 Thought] ◄── LLM 推理                环境状态变化
      ├─ 规划 Planning: 更新理解、拆任务                │
      └─ 工具选择 Tool Selection:                   [④ 新的 Observation]
          选哪个工具 + 定什么参数 ──────────────────────┘
                                          (反馈回①,进下一轮)
```

> 🔑 **行动不是终点。** 行动改变环境 → 环境吐出新观察 → 新观察被感知捕获 → 进入下一轮。**一个智能体就是从初始状态开始,靠不断转这个圈,一圈一圈逼近目标。**

这个循环像什么?像**你做饭**:看一眼菜谱(感知)→想下一步切什么(思考)→动手切菜(行动)→看切得怎么样(观察)→想接下来该炒了…一直转,直到菜出锅。

---

### 📜 1.2.3 TAO 协议:怎么让 LLM 明白"你要按这个格式来"

工程上让 LLM 驱动这个循环,靠的是一套**结构化输出协议**。LLM 每次回复不是自由聊天,而是固定格式:

```
Thought: 用户想知道北京天气,我需要调天气工具。      ← 慢思考/符号,可解释
Action:  get_weather("北京")                       ← 函数调用 = 符号指令
Observation: 北京当前晴,25℃,微风。                 ← 传感器把JSON翻译成"人话"
```

三个字段,各司其职:

| 字段 | 谁生成的 | 是什么 | 类比 |
|------|---------|--------|------|
| **Thought** | LLM 自己写的 | 决策的"快照"——我怎么分析当前情况、回顾上一步结果、下一步干嘛 | 你做题时在草稿纸上写的"先求x,再代入…" |
| **Action** | LLM 生成的函数调用 | 对外部世界的指令。被**Parser(解析器)**捕获,真去调函数 | 你说"Alexa,放音乐"——那是给机器的命令 |
| **Observation** | **感知系统**(不是 LLM!)生成的 | 工具返回原始数据(常是JSON)→感知系统把它**翻译成简洁的自然语言**再喂回 LLM | 助手把一长串原始数据帮你总结成一句话 |

> 🧠 **一个容易被误解的点**:Thought 是符号(LLM 写出来的结构化文字),Action 也是符号(精确的函数调用指令)。那"神经/快思考"到底在哪?——它**不在循环里任何一个看得见的步骤**,而是**藏在底层 LLM 引擎的一次前向传播里**。"我应该先查天气"这个直觉跳跃,是神经网络一次性完成的;但当它被写成 `Thought: …`,就已经**结晶成符号**了。**"生成即符号化"——这就是神经符号融合最精妙之处。**

> ⚠️ **另一个容易串台的概念**:反应式的"快"是**决策延迟**(感知到行动之间的时间),快思考的"快"是**知识是直觉的还是逻辑的**(知识存在形式)。**两个"快"在完全不同的坐标轴上,别混。**

## 1.3 动手体验:5 分钟实现第一个智能体

理论落地了。这一节用几十行 Python 把上面的 Agent Loop 和 TAO 协议**焊成能跑的代码**。

### 🧩 一个能跑的 Agent = 五个零件

| 理论概念 | 代码里的化身 | 干什么 | 类比 |
|---------|------------|--------|------|
| **TAO 协议/说明书** | `AGENT_SYSTEM_PROMPT` | 告诉 LLM:你是旅行助手/有这两个工具/**必须按 `Thought:/Action:` 格式输出**/收尾用 `Finish[...]` | 你给新员工发的《工作手册》 |
| **工具库(手脚)** | `get_weather()` + `get_attraction()` + `available_tools` 字典 | 真去互联网上调天气/搜景点;**工具内部把返回的 JSON 翻译成人话** | 员工桌上的电话和电脑 |
| **大脑** | `OpenAICompatibleClient.generate()` | 调用 LLM API,自主决策的源头。兼容任何 OpenAI 接口的服务 | 员工本人 |
| **Agent Loop** | `for i in range(5):` 主循环 | 感知→思考→行动→观察 四步的**实体化**。5 是安全上限,防死循环 | 工作流程:接任务→想→做→看结果→再接… |
| **记忆** | `prompt_history` 列表 | 每轮把整条 TAO 轨迹累积下来,再原封不动喂回给 LLM | 工作日志 |

### 🔁 主循环骨架:8 行代码 = 图 1.5 的逐字翻译

```python
prompt_history = [f"用户请求: {user_prompt}"]
for i in range(5):                                    # 最多5圈,防死循环
    full_prompt = "\n".join(prompt_history)           # ① 感知:历史拼成上下文
    llm_output  = llm.generate(full_prompt, SYS_PROMPT) # ② 思考:LLM吐 Thought+Action
    action = parse(llm_output)                          #    正则解析 Action
    if action.startswith("Finish"): break               #    收尾信号→退出
    observation = available_tools[tool](**kwargs)       # ③ 行动:真去调工具
    prompt_history.append(f"Observation: {observation}") # ④ 观察:塞回历史,进下一圈
```

> 🎯 **所有叫"智能体框架"的东西(LangChain/LlamaIndex/AutoGen…),内核就是这个 for 循环的工业加强版。** 到了第七章,你会亲手把这个简陋骨架升级成一套完整的框架。

### ▶️ 跑起来:三圈拆掉一个模糊指令

用户只说了一句话:"**查北京天气,然后根据天气推荐景点**"。注意——这是一个**两步依赖**任务:第二步(推荐)必须拿到第一步(天气)的结果才能做。

| 圈数 | Thought(思考) | Action(行动) | Observation(观察) |
|------|-------------|-------------|------------------|
| **1** | "先查天气" | `get_weather("北京")` | ☀️ 北京 Sunny,26℃ |
| **2** | "天气晴,据此推荐" | `get_attraction("北京", "Sunny")` | 🏯 颐和园、长城(←用到了圈1的结果!) |
| **3** | "信息够了,整合作答" | `Finish[...完整推荐...]` | ✅ 任务完成 |

四项基本能力全在这个三圈循环里:**任务分解**(一句话拆两步)→**工具调用**(调天气API+搜索API)→**上下文理解**(圈2用到了圈1的"Sunny")→**结果合成**(最后揉成一段完整人话)。

### 🎯 本节公式(建议背下来)

> **Agent = LLM(大脑) + Prompt(契约) + Tools(手脚) + Loop(循环) + History(记忆)**
>
> "智能"的本质 = 用 Prompt 给 LLM 套上 TAO 格式这张"契约" + 用循环把工具返回的观察不断喂回去。

## 1.4 智能体应用的协作模式

前三节把智能体内部剖开了。这一节换**外部视角**:智能体怎么跟我们、跟其他智能体一起干活?

### 1.4.1 Agent 作为"开发者工具":增强你,不取代你

Copilot、Cursor、Claude Code、Trae——它们深度嵌入开发流,自动化繁琐重复的部分(代码补全、重构、调试),**把创造性决策留给人**。

| 关系 | 模式 |
|------|------|
| 人 ↔ 工具 | **命令→执行**:你手把手指导每一步 |

---

### 1.4.2 Agent 作为"自主协作者":你定目标,它独立干活

这是真正的跳变:**关系从"命令-执行"升级为"目标-委托"。** 你把一个高层目标丢过去,Agent 自己规划→推理→执行→反思→交付。

| 关系 | 模式 |
|------|------|
| 人 ↔ 协作者 | **目标→委托**:你说"安排一次厦门之旅",它全套搞定 |

当前框架百花齐放,书里归纳为**三种架构范式**(= 全书后半段的路线图):

| 范式 | 核心思路 | 类比 | 代表框架 | 书里对应章节 |
|------|---------|------|---------|------------|
| **① 单智能体自主循环** | 一个 Agent 跑「思考-规划-执行-反思」闭环,自我提示、自我迭代 | 一个人包揽所有活:想→做→检查→改进→再做 | BabyAGI / AutoGPT | **四~五章**(ReAct/Plan-Solve/Reflection) |
| **② 多智能体协作** | 模拟人类团队:不同角色分工,对话协作 | 一个项目组:产品经理+程序员+测试各司其职 | CAMEL(角色扮演) / MetaGPT(公司SOP) / AutoGen(灵活网络) | **第六章** + **十三~十五**章 |
| **③ 高级控制流** | 把执行过程建模为**状态图**(State Graph),支持循环/分支/回溯/人工介入 | 地铁线路图:节点=状态,边=转换,可以绕圈/分叉/换乘 | LangGraph | **第六章 6.5** |

### 🔑 1.4.3 Workflow vs Agent:本章最重要的辨析

这是第一章最后一块、也是最有嚼劲的理论拼图:

> **Workflow = 让 AI 按部就班地执行你写好的剧本。**
> **Agent = 给 AI 一个目标,让它自己想办法达成。**

拿"报销审批"把二者放到同一场景里比:

| | 🗂️ Workflow(工作流) | 🤖 Agent(智能体) |
|---|---|---|
| **本质** | 一张精确、**静态**的流程图 | 一个自主、**动态**的目标驱动系统 |
| **决策怎么来的** | 写死的条件判断:`if 金额<500→经理批; else→+总监批` | LLM 根据实时信息**现场推理** |
| **遇意外怎么办** | 流程图里没画的 → **卡死** | 自己推理调整:天下雨?→推室内景点 |
| **自主性在哪** | ≈零(所有路径都是人预设的) | 高(LLM 现场推理,动态选路径) |
| **类比** | 流水线上的机械臂:每个动作写死在程序里 | 一个能干的助理:你说"安排出差",他自己去查航班/订酒店/排行程 |

> 🎯 金句:**没有任何一条 `if 天气=晴天 then 推荐颐和园` 的写死规则。** 如果下雨,Agent 会自己推理:雨天适合室内活动→推国家博物馆。它不是在"匹配规则",而是在**理解语义并推理**。

**但 Workflow 和 Agent 不是敌人,是搭档:**

| 场景 | 用谁 | 为什么 |
|------|------|--------|
| 金额审批、合规检查、固定流程 | **Workflow** | 确定性强、效率高、可审计 |
| 理解用户意图、动态排程、处理异常 | **Agent** | 需要语义理解和灵活推理 |

> 高手架构师不站队,而是**混搭**:确定性流程用 Workflow(又快又稳),模糊输入用 Agent(灵活推理)。LangGraph 就是这个思路的典范——用图(Workflow 思维)编排 Agent 节点,两者各取其长。
