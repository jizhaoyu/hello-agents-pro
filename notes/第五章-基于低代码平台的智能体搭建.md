## 1. 人类的记忆系统：我们的学习模板

### 1.1 人类记忆的三个层次

心理学研究发现，人类的记忆分为三个层次：

#### 📝 感觉记忆（Sensory Memory）

**是什么：** 感官接收到的信息的瞬时存储，持续不到1秒。

**类比：**
```
你走在街上，眼睛看到：
- 红色汽车驶过 (视觉)
- 有人喊了一声 (听觉)  
- 闻到咖啡香味 (嗅觉)

1秒后：大部分信息都忘了，只记得"刚才好像有辆车"
```

**特点：**
- 容量：巨大（所有感官信息）
- 持续时间：极短（<1秒）
- 作用：过滤掉无关信息

#### 🧠 工作记忆（Working Memory / 短期记忆）

**是什么：** 当前正在处理的信息，像大脑的"RAM"。

**经典实验：** 心理学家发现，人类的工作记忆大约能记住**7±2个项目**。

**例子：**
```
我给你一串数字：
8 - 3 - 6 - 1 - 9 - 4 - 7 - 2 - 5

现在闭上眼睛，试着回忆...

大多数人只能记住5-9个数字。
```

**工作记忆在干什么：**
```
场景：你在做数学题

工作记忆：
- 题目要求什么？ (当前任务)
- 上一步算出了什么？ (中间结果)
- 接下来要算什么？ (下一步计划)

如果信息太多，工作记忆就"卡"了 → 你需要拿张草稿纸记下来
```

**特点：**
- 容量：有限（7±2个项目）
- 持续时间：短暂（几秒到几分钟）
- 作用：当前任务的"工作台"

#### 💾 长期记忆（Long-term Memory）

**是什么：** 持久存储的知识和经验，像大脑的"硬盘"。

**例子：**
```
问：你的生日是哪天？
答：[立刻想起] 1995年8月15日

问：小学一年级的班主任叫什么？
答：[努力回忆...] 好像是王老师？

问：2年前的今天你在哪？
答：[想不起来] 完全不记得了...
```

**长期记忆的特点：**
- 容量：理论上无限
- 持续时间：可能终身
- 但不是所有信息都能存进去！

**什么信息会进入长期记忆？**
1. **重复多次**：你的名字（重复了几万次）
2. **情感强烈**：初恋的名字（情感记忆深刻）
3. **有意识记忆**：考试前背的公式
4. **关联强的信息**：和你兴趣相关的内容

### 1.2 人类记忆系统的工作流程

```
外界刺激
   ↓
感觉记忆 (筛选)
   ↓ (注意力)
工作记忆 (处理当前任务)
   ↓ (重复/重要性)
长期记忆 (永久存储)
```

**完整例子：学习一个新单词**

```
1. 感觉记忆阶段：
   你看到单词 "serendipity"
   眼睛捕获这个词的视觉信息

2. 工作记忆阶段：
   大脑：这个词怎么读？什么意思？
   查词典：[ser-en-dip-i-ty] 意外发现美好事物的能力
   尝试造句："It was pure serendipity..."

3. 长期记忆阶段：
   - 如果只看一次 → 很快忘记
   - 如果遇到5次 → 开始记住
   - 如果用在作文里 → 深刻记忆
```

## 2. 智能体的记忆系统：模仿人类

### 2.1 为什么不能把所有对话都给LLM？

**问题：上下文窗口限制**

LLM的上下文窗口是有限的：
- GPT-3.5：4k tokens（约3000个汉字）
- GPT-4：8k-128k tokens
- Claude-3：200k tokens

**实际例子：**
```
假设你的Agent的上下文是8k tokens：

1. 系统提示词：1000 tokens
2. 工具描述：500 tokens
3. 对话历史：？？？ tokens
4. 当前问题：100 tokens

对话历史能用的空间：8000 - 1000 - 500 - 100 = 6400 tokens

如果一轮对话平均200 tokens：
6400 ÷ 200 = 32轮对话

第33轮开始，就装不下了！
```

**如果强行装下会怎样？**
- 成本飙升：每个token都要付费
- 速度变慢：处理的内容越多越慢
- 效果下降：信息太多，LLM反而抓不住重点

**解决方案：记忆系统**

不是把所有历史都给LLM，而是：
1. 只给"最近的对话"（短期记忆）
2. 只给"重要的信息"（长期记忆）
3. 根据当前问题"检索相关记忆"

### 2.2 智能体记忆的三层架构

我们的实现模仿了人类的记忆系统：

```
┌─────────────────────────────────────────┐
│   工作记忆 (Working Memory)              │
│   - 当前任务                             │
│   - 中间结果                             │
│   - 临时上下文                           │
│   持续时间：一个任务的生命周期            │
└─────────────────────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│   短期记忆 (Short-term Memory)           │
│   - 最近N轮对话                          │

│   - 自动淘汰旧对话                       │
│   持续时间：一个会话                     │
└─────────────────────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│   长期记忆 (Long-term Memory)            │
│   - 用户信息（姓名、偏好）               │
│   - 重要事实                             │
│   - 历史摘要                             │
│   持续时间：跨会话持久化                 │
└─────────────────────────────────────────┘
```

## 3. 工作记忆（Working Memory）详解

### 3.1 什么是工作记忆？

**定义：** 智能体在执行当前任务时的临时"工作台"。

**类比：做饭时的厨房台面**
```
你在做宫保鸡丁：

台面上放着：
- 菜谱（任务描述）
- 切好的鸡肉（中间结果1）
- 处理好的花生（中间结果2）
- 调好的酱汁（中间结果3）
- 接下来要炒了（下一步计划）

做完这道菜后：
台面清理干净，准备做下一道菜
```

### 3.2 工作记忆存什么？

**代码示例解析：**
```python
class WorkingMemory:
    def __init__(self):
        self.current_task: str = ""           # 当前任务
        self.current_context: Dict = {}       # 任务上下文
        self.intermediate_results: List = []  # 中间结果
```

**实际例子：**
```python
# 任务：帮用户规划北京三日游

工作记忆内容：
{
    "current_task": "规划北京三日游",
    "current_context": {
        "destination": "北京",
        "days": 3,
        "start_time": "2026-08-01"
    },
    "intermediate_results": [
        "已查询天气：晴天",
        "已搜索景点：故宫、长城、颐和园",
        "已预订酒店：北京饭店"
    ]
}

任务完成后 → 工作记忆清空 → 准备下一个任务
```

### 3.3 为什么需要工作记忆？

**场景对比：**

**没有工作记忆：**
```
用户："帮我规划北京三日游"

Agent:
- Step 1: 查天气 → 忘了要干什么了
- Step 2: ？？？ → 不知道接下来该做什么
```

**有工作记忆：**
```
用户："帮我规划北京三日游"

Agent工作记忆：
- 任务：规划北京三日游
- 已完成：查天气（晴）
- 下一步：根据天气选景点

Agent:
- Step 1: 查天气 → 记录结果
- Step 2: [看工作记忆] 天气晴，推荐户外景点
- Step 3: [看工作记忆] 已有景点，开始规划路线
```

## 4. 短期记忆（Short-term Memory）详解

### 4.1 什么是短期记忆？

**定义：** 保存最近N轮对话的"滑动窗口"。

**类比：聊天软件的消息记录**
```
微信对话框：

[向上滚动，可以看历史消息]
...
你：晚上吃什么？
朋友：火锅吧
你：好的，几点？
朋友：7点
你：[当前输入] 在哪见？

这些都是"短期记忆" - 你们刚才说了什么
```

### 4.2 滑动窗口机制

**为什么叫"滑动窗口"？**

```
假设短期记忆最多保留5轮对话：

初始状态：
┌────────────┐
│ [空]       │ ← 窗口
│ [空]       │
│ [空]       │
│ [空]       │
│ [空]       │
└────────────┘

第1轮对话：
┌────────────┐
│ 对话1      │ ← 窗口
│ [空]       │
│ [空]       │
│ [空]       │
│ [空]       │
└────────────┘

第5轮对话：
┌────────────┐
│ 对话1      │ ← 窗口已满
│ 对话2      │
│ 对话3      │
│ 对话4      │
│ 对话5      │
└────────────┘

第6轮对话（关键时刻！）：
┌────────────┐
│ 对话2      │ ← 对话1被挤出去了
│ 对话3      │   窗口"滑动"了
│ 对话4      │
│ 对话5      │
│ 对话6      │ ← 新对话进来
└────────────┘
```

**代码实现：**
```python
from collections import deque

class ShortTermMemory:
    def __init__(self, max_turns: int = 10):
        # deque 是双端队列，自动实现滑动窗口
        self.conversations = deque(maxlen=max_turns)
    
    def add_turn(self, user_msg: str, agent_msg: str):
        turn = {
            "timestamp": "2026-07-14 10:30:00",
            "user": user_msg,
            "agent": agent_msg
        }
        # 当队列满时，自动移除最老的元素
        self.conversations.append(turn)
```

### 4.3 为什么只保留最近的对话？

**原因1：上下文窗口限制**
```
如果保留所有对话：
第1轮：200 tokens
第2轮：200 tokens
...
第100轮：20,000 tokens ← 超出窗口了！
```

**原因2：成本控制**
```
每次调用LLM：
输入tokens × 单价 = 成本

如果每次都传100轮对话：
20,000 tokens × $0.001 = $0.02 每次
一天1000次调用 = $20

如果只传10轮对话：
2,000 tokens × $0.001 = $0.002 每次
一天1000次调用 = $2

省了90%的成本！
```

**原因3：更相关的上下文**
```
对话历史：
第1轮（昨天）："今天天气怎么样？"
第2轮（昨天）："推荐个餐厅"
...
第50轮（现在）："刚才你推荐的餐厅在哪？"

LLM只需要看最近的对话，
第1轮的天气查询已经不相关了。
```

### 4.4 短期记忆的实际运作

**完整例子：**

```python
# 创建短期记忆（最多5轮）
memory = ShortTermMemory(max_turns=5)

# 第1-5轮对话
memory.add_turn("你好", "你好！")
memory.add_turn("天气如何", "今天晴天")
memory.add_turn("推荐景点", "可以去公园")
memory.add_turn("公园在哪", "在市中心")
memory.add_turn("怎么去", "坐地铁2号线")

# 查看短期记忆
print(memory.get_recent_history())
# 输出：5轮对话

# 第6轮对话（超出容量）
memory.add_turn("要门票吗", "免费开放")

# 再查看短期记忆
print(memory.get_recent_history())
# 输出：还是5轮对话
# 但第1轮"你好"已经被移除了！
```

## 5. 长期记忆（Long-term Memory）详解

### 5.1 什么是长期记忆？

**定义：** 持久化存储的重要信息，跨会话保持。

**类比：你的通讯录 + 笔记本**
```
通讯录：
- 朋友的名字、电话（事实性信息）
- 朋友的生日、爱好（个人信息）

笔记本：
- 重要会议的摘要
- 关键决策的记录
- 经验教训的总结

这些信息：
- 不会因为你睡一觉就忘记（持久化）
- 可以随时查阅（检索）
- 会影响你未来的决策（应用）
```

### 5.2 长期记忆存什么？

我们的实现包含三类信息：

#### 📋 事实性知识（Facts）

**是什么：** 客观的、不变的信息。

**例子：**
```python
facts = {
    "user_name": {
        "value": "张三",
        "timestamp": "2026-07-14 10:00:00",
        "access_count": 5  # 被访问了5次
    },
    "user_birthday": {
        "value": "1995-08-15",
        "timestamp": "2026-07-14 10:05:00",
        "access_count": 1
    },
    "user_company": {
        "value": "字节跳动",
        "timestamp": "2026-07-14 10:10:00",
        "access_count": 3
    }
}
```

**为什么要记录访问次数？**
```
访问次数反映了信息的重要性：

access_count = 100 → 非常重要，经常用到
access_count = 1   → 可能不太重要

未来可以实现"遗忘机制"：
- 长期不访问的信息 → 降低优先级
- 或者自动归档、删除
```

#### ❤️ 用户偏好（Preferences）

**是什么：** 用户的喜好、习惯、倾向。

**为什么单独存储？**
```
事实 vs 偏好：

事实："用户的名字是张三"
     → 客观的，不会变（除非改名）

偏好："用户喜欢 Python 编程"
     → 主观的，可能会变（兴趣转移）
     → 需要定期更新
```

**例子：**
```python
preferences = {
    "programming_language": "Python",
    "communication_style": "简洁直接",
    "working_hours": "早上9点-晚上6点",
    "favorite_food": "川菜",
    "learning_style": "喜欢通过实例学习"
}

# 用偏好影响 Agent 的行为：
if preferences["communication_style"] == "简洁直接":
    # Agent 的回答应该简短有力
    response = "好的，已完成。"
else:
    # Agent 的回答可以详细一些
    response = "好的，我已经完成了您的请求。具体来说..."
```

#### 📝 历史摘要（History Summaries）

**是什么：** 过去对话的压缩版本。

**为什么需要摘要？**
```
问题：短期记忆只保留最近10轮对话

假设今天有100轮对话：
- 短期记忆：只有最近10轮
- 前90轮：完全丢失了！

但是，前90轮可能有重要信息：
第20轮："我对机器学习很感兴趣"
第45轮："我在学习深度学习"

解决方案：定期压缩成摘要
```

**摘要的生成：**
```python
# 原始对话（前10轮）
对话1: "介绍一下Python"
对话2: "Python有什么用"
对话3: "如何学习Python"
对话4: "推荐Python书籍"
对话5: "Python和Java的区别"
...

# 压缩成摘要
摘要: "用户咨询了Python编程相关的问题，包括基础介绍、
      应用场景、学习资源和与其他语言的对比。"

# 摘要只占原来1/10的空间，但保留了核心信息
```

### 5.3 长期记忆的检索

**关键问题：怎么从长期记忆中找到相关信息？**

#### 方法1：关键词匹配（简单版）

```python
def search_relevant(self, query: str) -> List[str]:
    """搜索相关记忆"""
    relevant = []
    
    # 在事实中搜索
    for key, data in self.facts.items():
        if query in key or query in str(data["value"]):
            relevant.append(f"事实: {key} = {data['value']}")
    
    # 在摘要中搜索
    for summary in self.history_summary:
        if query in summary["summary"]:
            relevant.append(f"历史: {summary['summary']}")
    
    return relevant
```

**例子：**
```python
用户问："我上次学的是什么？"

检索过程：
1. 提取关键词："学"
2. 在事实中搜索：
   - user_learning_topic: "Python编程" ✓ 匹配
3. 在摘要中搜索：
   - "用户咨询了Python编程..." ✓ 匹配

返回：
- "事实: user_learning_topic = Python编程"
- "历史: 用户咨询了Python编程..."
```

#### 方法2：语义匹配（高级版，需要向量数据库）

**关键词匹配的局限：**
```
用户问："我想学的那个编程语言是啥来着？"

关键词："学"、"编程语言"

但长期记忆中存的是：
- user_learning_topic: "Python"

关键词匹配：找不到！
因为"Python"和"编程语言"字面上不匹配
```

**语义匹配的优势：**
```
1. 把所有记忆转换成向量（embeddings）
   "Python" → [0.2, 0.8, 0.3, ...]
   "编程语言" → [0.25, 0.78, 0.31, ...]
   
2. 计算相似度
   cosine_similarity([0.2, 0.8, 0.3], [0.25, 0.78, 0.31])
   = 0.95 (非常相似！)
   
3. 返回最相似的记忆
```

**未来优化方向：**
```python
# 使用向量数据库（如 Faiss、Chroma）
from chromadb import Client

client = Client()
collection = client.create_collection("long_term_memory")

# 存储记忆
collection.add(
    documents=["用户的学习主题是Python编程"],

)

# 语义检索
results = collection.query(
    query_texts=["我想学的那个编程语言是啥"],
    n_results=3
)
# 能正确找到 "Python编程"！
```

### 5.4 长期记忆的持久化

**为什么需要持久化？**
```
场景1：用户关闭浏览器
- 没有持久化：所有记忆丢失
- 有持久化：下次打开还记得

场景2：服务器重启
- 没有持久化：所有用户的记忆丢失
- 有持久化：服务恢复后记忆还在
```

**实现方式：JSON文件**

```python
def save_to_file(self, filepath: str):
    """保存到文件"""
    data = {
        "facts": self.long_term.facts,
        "preferences": self.long_term.preferences,
        "summaries": self.long_term.history_summary
    }
    with open(filepath, 'w', encoding='utf-8') as f:
        json.dump(data, f, ensure_ascii=False, indent=2)

def load_from_file(self, filepath: str):
    """从文件加载"""
    with open(filepath, 'r', encoding='utf-8') as f:
        data = json.load(f)
        self.long_term.facts = data.get("facts", {})
        self.long_term.preferences = data.get("preferences", {})
        self.long_term.history_summary = data.get("summaries", [])
```

### 2.2 工具参数的定义：ToolParameter

**为什么需要定义参数？**

```python
@dataclass
class ToolParameter:

    required: bool     # 是否必需
    default: Any       # 默认值

```

**实际使用：**
```python

# 验证用户输入
is_valid, error, converted = param.validate("100+200")

# converted = "100+200"
```

### 2.3 参数验证的魔法：自动类型转换

**问题：LLM生成的参数可能类型不对**

```python
# LLM可能生成：
search_web(query=["Python", "教程"])  # 传了列表
search_web(query=123)                 # 传了数字
search_web(max_results="5")           # 传了字符串的"5"而不是数字5
```

**我们的框架会自动转换：**
```python
def validate(self, value):
    """验证并转换参数"""
    
    if self.type == "str":
        # 任何类型都转成字符串
        converted = str(value)
        # ["Python", "教程"] → "['Python', '教程']"
        # 123 → "123"
    
    elif self.type == "int":
        # 转成整数
        converted = int(value)
        # "5" → 5
        # 5.0 → 5
    
    elif self.type == "float":
        converted = float(value)
        # "3.14" → 3.14
        # 3 → 3.0
    
    elif self.type == "bool":
        # 智能转换布尔值
        if isinstance(value, bool):
            converted = value
        else:
            # "true", "1", "yes" → True
            # "false", "0", "no" → False
            converted = str(value).lower() in ['true', '1', 'yes']
    
    return converted
```

**例子：**
```python
# 定义参数
max_results = ToolParameter(
    name="max_results",
    type="int",
    required=False,
    default=5
)

# 各种输入都能处理
max_results.validate("10")   # → 10 (int)
max_results.validate(10)     # → 10 (int)
max_results.validate(10.0)   # → 10 (int)
max_results.validate("abc")  # → 报错：无法转换成int
max_results.validate(None)   # → 5 (使用默认值)
```

## 3. 装饰器模式：优雅的工具注册

### 3.1 什么是装饰器？

**最简单的理解：装饰器 = 给函数"穿衣服"**

**类比：咖啡店**
```
基础咖啡：
def coffee():
    return "咖啡"

加糖：
def add_sugar(coffee_func):
    def wrapper():
        return coffee_func() + " + 糖"
    return wrapper

加奶：
def add_milk(coffee_func):
    def wrapper():
        return coffee_func() + " + 奶"
    return wrapper

# 使用装饰器
@add_sugar
@add_milk
def my_coffee():
    return "咖啡"

print(my_coffee())
# 输出："咖啡 + 奶 + 糖"

# 相当于：
# my_coffee = add_sugar(add_milk(coffee))
```

### 3.2 @tool 装饰器的工作原理

**不用装饰器（繁琐）：**
```python
# 1. 定义工具函数
def search_web(query):
    return f"搜索：{query}"

# 2. 手动创建元数据
metadata = ToolMetadata(
    name="search_web",
    description="搜索网络",
    parameters=[
        ToolParameter("query", "str", required=True)
    ]
)

# 3. 手动注册
tool_manager.register(search_web, metadata)
```

**用装饰器（优雅）：**
```python
@tool(
    name="search_web",
    description="搜索网络",
    parameters=[
        ToolParameter("query", "str", required=True)
    ]
)
def search_web(query):
    return f"搜索：{query}"

# 装饰器自动：
# 1. 创建元数据
# 2. 附加到函数上
# 3. 等待注册
```

**装饰器的实现：**
```python
def tool(name=None, description="", parameters=None):
    """工具装饰器"""
    
    def decorator(func):
        # 创建元数据
        metadata = ToolMetadata(
            name=name or func.__name__,
            description=description or func.__doc__,
            parameters=parameters or []
        )
        
        # 把元数据"贴"在函数上
        func._tool_metadata = metadata
        
        # 返回原函数（功能不变，只是添加了元数据）
        return func
    
    return decorator
```

**使用效果：**
```python
# 定义工具
@tool(name="search", description="搜索网络")
def search_web(query):
    return f"搜索：{query}"

# 函数功能不变
result = search_web("Python")
print(result)  # "搜索：Python"

# 但现在函数"知道"自己是个工具了
print(search_web._tool_metadata.name)  # "search"
print(search_web._tool_metadata.description)  # "搜索网络"
```

## 6. 执行统计：可观测性

```python
@tool(
```

```python
manager = ToolManager()
```
